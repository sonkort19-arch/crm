# Чеклист перед изменениями

Перед коммитом или после правок в CRM пройти пункты.

## Разметка и события

- [ ] **HTML `id`** совпадает с тем, что используется в `getElementById` / `querySelector` в JS.
- [ ] **addEventListener** добавлен для новых кнопок/полей; при удалении элементов — нет «висячих» ссылок.
- [ ] **Функция** вызывается из нужного места (сервис / render / events) и **существует** в модуле.

## Данные и отрисовка

- [ ] После изменения данных на сервере клиент вызывает **`loadUsersFromServer()`** (внутри сервисов после успешного `fetch`).
- [ ] После изменения данных вызывается **`render`** (или конкретный `renderUsers` / `renderDetails` / `renderSettings` / `refreshTotalBalance`), чтобы UI не «отставал».

## Авторизация

- [ ] Новые защищённые маршруты используют **`requireAuth`** и при необходимости **`requireRole('admin')`** на сервере.
- [ ] Клиентский **`api.js`** передаёт **`Authorization: Bearer`** для запросов к `/api/users`, `/api/logs`, операциям.
- [ ] Скрытие кнопок по роли не заменяет проверку на сервере.

## Качество

- [ ] **Нет ошибок в консоли** браузера при открытии приложения и при основных сценариях (вход → список → карточка → настройки / начисления).
- [ ] **ES modules:** приложение открывается через HTTP (не `file://`), если браузер блокирует модули.

## Архитектура

- [ ] Бизнес-логика не попала в `ui/render.js` без необходимости.
- [ ] Сервисы не обращаются к `document`.

Дополнительно: см. `architecture.md`, `code-rules.md`, `data-rules.md`, `ui-rules.md`.
